Hostinger layout (this project)
================================

DEPLOY BUNDLE FOR WEB ROOT
--------------------------
  idea/_public_html

This folder is a full copy of Laravel’s `public/` folder (generated by
`php sync-public-to-public-html.php` in this directory). Treat it as what
you upload INTO Hostinger **public_html** (merge or replace files).


DIRECTORY SHAPE ON THE SERVER (required by index.php)
-----------------------------------------------------
Your default `public_html/index.php` expects the Laravel app ONE level
above `public_html`:

  .../your-domain-root/
      vendor/              ← from project root (not inside public_html)
      app/
      bootstrap/
      config/
      database/
      resources/
      routes/
      storage/
      .env
      artisan
      public_html/         ← upload the *contents* of idea/_public_html here
          index.php
          .htaccess
          assets/
          ...

Paths in index.php:
  ../vendor/autoload.php
  ../bootstrap/app.php
  ../storage/framework/maintenance.php


FULL APP INSIDE public_html (flat / “everything in web root”)
-------------------------------------------------------------
If you uploaded the entire Laravel tree INTO public_html (app/, vendor/,
bootstrap/, index.php, etc. side by side), the default public/index.php
paths are WRONG: ../vendor points above public_html and the site will not
boot.

Use the alternate front controller from this repo (same paths, all under
__DIR__):

  public/index.webroot-same-dir.php

On the server:
  1) Back up:  cp index.php index.php.bak
  2) Copy the contents of index.webroot-same-dir.php over index.php, or
     replace index.php with that file renamed to index.php.

bootstrap/app.php still resolves the application base path to the directory
that contains the bootstrap folder (your public_html root), so app boot is
correct. Prefer the “domain root + public_html = assets only” layout above
when you can; use this when Hostinger only allows a single document root.


RE-SYNC WEB ASSETS LOCALLY
--------------------------
From the `idea` folder:

  php sync-public-to-public-html.php

Then upload `idea/_public_html` again (or only changed files).


PHP / BLADE CHANGES
-------------------
Anything outside `public/` (controllers, `resources/views`, routes, etc.)
must be deployed to the **domain root** next to `public_html`, not inside
`_public_html`. Uploading only `_public_html` updates CSS/JS/static files;
template and backend changes need the Laravel tree updated too.


OPTION
------
Point the domain’s document root to `project/public` and deploy the full
`idea` tree via Git/FTP; then you do not need a separate `_public_html` copy.


TROUBLESHOOTING: “Unsafe attempt to load URL … from frame … chrome-error://”
----------------------------------------------------------------------------
That message is almost never your Blade/JS code. It usually means the browser
failed earlier (SSL, DNS, 500 error, wrong document root). Chrome’s error
page cannot put your HTTPS site inside an iframe.

Check in order:
  1) hPanel → SSL → enable Let’s Encrypt for ideahhub.com (or your exact host);
     wait until the certificate is Active.
  2) Open EXACTLY: https://ideahhub.com — if you see “Not secure” or a cert
     warning, fix SSL before debugging Laravel.
  3) .env on server: APP_URL=https://ideahhub.com (https, no typo; match www
     vs non-www to your domain settings).
  4) APP_ENV=production forces HTTPS in AppServiceProvider — if SSL is not
     working yet, temporarily use APP_ENV=local only to diagnose, then fix SSL.
  5) Deploy full Laravel next to public_html (vendor/, app/, …). Only
     public_html without ../vendor will white-screen or error.
  6) storage/ and bootstrap/cache/ writable (755/775 as Hostinger recommends).
  7) TrustProxies: this repo sets trusted proxies for Hostinger so Laravel
     sees HTTPS behind their proxy (upload updated app/ after git pull).

If it still fails: Hostinger → Advanced → PHP error log, or enable APP_DEBUG=true
briefly to read the real exception (turn off in production).
